## What is this?

`temple` is an always-on, server/client AI agent harness. **renco** is the
agent — a persistent coding assistant that lives on oracle (an always-on M2
Mac mini) and talks to models hosted across your GPU fleet. The agent reaches
your workstations over SSH through the bastion, so tools run on your actual
machines, not on the server.

```
┌─────────────────────────────────────────────────────────┐
│               temple-server (oracle, daemon)             │
│  • WebSocket API + token auth                            │
│  • Agent loop with remote tool use (files, shell, SSH)   │
│  • MCP tools via litellm (fetch, searxng, nixos, arxiv,  │
│    context7) — run server-side                            │
│  • Permission system (per-session, user-prompted)        │
│  • SQLite memory + skills + personality                  │
│  • Planner/executor/reviewer pipeline (deepseek + qwen)  │
│  • Signal bot (two-way via signal-cli)                   │
│  • Cron: skills extraction, smart flake updates          │
└──────────────────────────────────────────────────────────┘
        ▲ WebSocket (JSON)               ▲ SSH (bastion)
   (TUI clients)                   (tool execution)
┌───────┴────────┐              ┌───────┴────────┐
│ temple (TUI)   │              │ your workstation │
│ iocraft, tools │              │ e-desktop, ...   │
│ run locally    │              │ actual files     │
└────────────────┘              └────────────────┘
        │                              ▲
        ▼                              │ Signal
   litellm proxy → models          (phone)
   (deepseek on son-of-anton,
    qwen on anton,
    gemma on anton/e-desktop,
    classifier on oracle)
```

## Quick start

```bash
nix build                    # both binaries in result/bin/
result/bin/temple-server     # daemon (needs LITELLM_API_KEY or LITELLM_MASTER_KEY)
result/bin/temple            # TUI client, another terminal
```

## Client commands

| Input | Action |
|---|---|
| `/sessions` | list persisted sessions |
| `/session N` | resume session by index or id prefix |
| `/new [target] [dir]` | new session, optionally SSH + start dir |
| `/models` | list models from litellm |
| `/model NAME` | switch session model |
| `/mode MODE` | permission mode: `default` `ask` `lockdown` `yolo` |
| `/help` | help |
| `:q` | quit |
| `↑` `↓` | input history recall |
| `Shift+Tab` | cycle permission mode |
| `Ctrl+G` | open prompt in `$EDITOR` |
| `Ctrl+L` | clear chat |
| `PgUp/PgDn`, `Ctrl+U/D` | scroll |

The current permission mode is always visible in the status bar
(`[default]`, `[ask]`, `[lockdown]`, `[YOLO]`). The client also enforces
its own consent layer: writes outside the working directory and non-safe
shell commands prompt locally with y/N, regardless of the server's mode.

Each assistant reply gets a stats footer:

```
renco › fixed the borrow error in main.rs.
       ⏱ qwen3.6-27b-coding · 41.2 tok/s · 12.4k ctx · 12.4k+890 tok · 21.6s
```

For multi-step work the agent maintains a persistent task list (the `todo`
tool) — visible as a checklist in the TUI and on Signal, and restored on
session resume.

## Signal bot

Text renco on Signal — the bot runs via signal-cli daemon on server-mu.
Commands work the same as the TUI:

```
/sessions              → list recent sessions
/session <prefix>      → resume a session
/new e-play tls        → new session in ~/tls on e-play@e-desktop
/quick                 → back to quick mode
/mode [mode]           → show or set permission mode
/help                  → all commands
```

Responses get a preamble showing the active session and its permission mode:
```
📋 e-play@e-desktop · a1b2c3 · fixing the parser [ask]

list the files in this directory...
```

Code blocks arrive as Signal monospace blocks, tables keep their column
alignment, and task lists render as checklists. Read receipts, typing
bubbles, and "still consulting the oracle..." status updates keep you
informed while renco works.

## NixOS module

```nix
# flake.nix
inputs.temple.url = "github:ewtodd/temple";

# configuration.nix
imports = [ inputs.temple.nixosModules.temple-server ];

  allowedDirs = [ "/etc/nixos" "/home" ];
  signal.enable = true;
  sshBastion = "bastion";
  sshTargets = [
    { name = "e-work@e-desktop"; account = "e-work"; host = "10.0.0.4"; port = 2222; owner = "ethan"; }
    { name = "e-play@e-desktop"; account = "e-play"; host = "10.0.0.4"; port = 2222; owner = "ethan"; }
  ];
};
```

The module generates a complete config.toml, runs the daemon as a hardened
systemd service (dedicated `temple` user, `ProtectSystem=full`, state in
`/var/lib/temple`), generates an SSH config for the temple user, and opens
the firewall port if requested.

Secrets (API keys, tokens, phone numbers) come from agenix via
`environmentFile`.

## Architecture notes

- **Server**: Rust, tokio, WebSocket JSON protocol. Runs on oracle (M2 Mac
  mini, Asahi, aarch64). Sessions are SQLite-backed and persist across
  connections — resume from TUI, continue from Signal, pick up from another
  host.
- **Tool execution**: local tools (`read_file`, `write_file`, `list_dir`,
  `execute_command`) execute on the TUI client's machine via ToolRequest/
  ToolResult messages over WebSocket. SSH-backed sessions execute on the
  workstation via the bastion relay (wake-on-LAN supported). MCP tools
  (fetch, searxng, nixos, arxiv, context7) and `save_memory` run on oracle.
- **Router**: heuristic classifier determines model routing. Simple →
  researcher model (gemma-4-31b). Medium → default model (deepseek).
  Complex → planner→executor→reviewer pipeline (deepseek decomposes, qwen
  executes, deepseek reviews, max 3 revision rounds). Critical → deepseek
  direct. System prompt is tiered: Quick sessions (Signal) get minimal
  prompt with MCP tools and date; Coding sessions get full code rules +
  project context.
- **Signal bot**: two-way via signal-cli JSON-RPC daemon on server-mu
  (x86_64 bastion — signal-cli's native lib doesn't support aarch64).
  Read receipts, typing bubbles, "still consulting the oracle..." status,
  multi-message splitting.
- **Auth**: Bearer token auth on WebSocket. Token file lines are
  `token:person:phone[:admin[:priority]]` (`admin` = `yes` unlocks
  /broadcast; `priority` is the queue tier — lower runs first: 0 ethan,
  1 valarie, -1 default). Generate with
  `temple-server --generate-token USER PHONE [--admin] [--priority N]`.
- **Queue**: one agent loop runs at a time across all sessions (Signal,
  TUI). Queued requests dequeue by user priority, non-preemptive — ethan's
  work never waits behind anyone else's queued requests.
- **Sessions**: owned by the token identity (e.g. "ethan"), not the
  computer account. OpenSession auto-matches hostname+username to SSH
  targets. `--continue` resumes the most recent session.
- **Keep-alive**: pings the primary model every 30s to keep llamaswap
  resident.
- **Cron**: 03:00 skills extraction (model call → `skills.md`), 04:00
  smart `nix flake update` (GitHub compare API, reverts if risky),
  Sunday 05:00 personality self-update — all local time, with per-job
  catch-up so a missed window still runs once that day. All events notify
  Signal users.
